MH1 / MPOC / MPIC Processor

This chapter documents the Cloud Run workflow used to retrieve and publish MODIS-Aqua Level-3 satellite products for the MH1, MPOC, and MPIC datsets served on ERDDAP. The workflow handles both near-real-time (NRT) and science-quality streams for ocean color, sea surface temperature, particulate organic carbon, and particulate inorganic carbon products.

The workflow replaces legacy server-based retrieval scripts with a containerized cloud workflow. It queries the NASA OceanColor file search API, compares available files against products already staged in Google Cloud Storage (GCS), downloads only missing files, publishes final products to ERDDAP-facing storage paths, and removes temporary files from the Cloud Run runtime.

Purpose

The processor manages three related MODIS-Aqua product families:

  • MH1 Ocean Color — chlorophyll-a, nFLH, PAR, and Kd490
  • MH1 SST — raw sea surface temperature plus a masked SST variant that retains pixels where qual_sst >= 0
  • MPOC — particulate organic carbon
  • MPIC — particulate inorganic carbon

Each product stream supports 1-day, 8-day, and monthly composites where available. NRT and science-quality products are handled separately so that ERDDAP can expose both recent operational products and delayed higher-quality products.

Cloud Run job

The workflow runs as a scheduled Cloud Run job. The job entrypoint prepares writable runtime directories under /tmp, copies Earthdata credential files from Secret Manager mounts, and then runs the retrieval suites in sequence: NRT products first, followed by science-quality products.

Suite Scripts Description
MH1 NRT getMH1OceanColor_NRT.py, getMH1SST_NRT.py Ocean color and SST near-real-time retrieval
MH1 Science getMH1OceanColor_Sci.py, getMH1SST_Sci.py Ocean color and SST science-quality retrieval
MPIC NRT getMPIC_NRT.py PIC near-real-time retrieval
MPIC Science getMPIC_Sci.py PIC science-quality retrieval
MPOC NRT getMPOC_NRT.py POC near-real-time retrieval
MPOC Science getMPOC_Sci.py POC science-quality retrieval

Processing pipeline

Each retrieval suite follows the same operational pattern:

Cloud Scheduler
  → Cloud Run Job
  → Query NASA OceanColor file search API
  → Compare available NASA files against existing GCS objects
  → Download missing files to /tmp
  → Publish outputs to configured GCS destinations
  → Remove local temporary files

The NASA OceanColor file_search API is queried for the current calendar year. Returned filenames are compared against the production bucket inventory from list_bucket_content(). Only files absent from GCS are downloaded using Earthdata credentials provided at runtime. This keeps the job incremental and avoids repeatedly downloading products that have already been staged.

Runtime configuration

Runtime settings are loaded from config/config.yml through roylib.py. In Cloud Run, the config file is mounted from Secret Manager and the path is provided through the ROYLIB_CONFIG environment variable.

The workflow distinguishes between two major GCS destinations:

Setting Purpose
ERDPROD_BUCKET Production bucket for final ERDDAP-facing NetCDF products
ERDWORK_BUCKET Work/staging bucket for intermediate, mirrored, or operational support files

The standard publishing logic is implemented in send_to_servers():

  1. Every output is uploaded to the production bucket.
  2. Selected 1-day MB/MW outputs are also copied to the work bucket using the legacy regional layout.

Example destination patterns:

gs://<prod_bucket>/<prod_root>/<dst_dir>/<interval>day/<filename>
gs://<work_bucket>/<work_root>/<region>/1day/<filename>

Bucket values in config.yml are plain bucket names, not gs:// URIs.

The configuration also provides:

  • NASA OceanColor API endpoints
  • ERDDAP-facing publish destinations
  • local runtime paths used inside the container
  • Earthdata credential file locations
  • product-specific destination mappings

Some configuration files may contain settings shared with other CoastWatch workflows. The MH1 processor uses the subset related to MODIS-Aqua MH1, MPOC, MPIC, NASA file search, and GCS publishing.

NASA API dataset IDs

The scripts query NASA OceanColor using dataset identifiers (dtid) for MODIS-Aqua Level-3 mapped products. The configured queries use sensor_id=7, corresponding to MODIS-Aqua.

Dataset dtid Notes
MH1 Ocean Color NRT 1055 MODIS-Aqua Level-3 mapped NRT ocean color products
MH1 Ocean Color Science 1043 MODIS-Aqua Level-3 mapped science-quality ocean color products
MH1 SST NRT 1061 MODIS-Aqua Level-3 mapped NRT SST products
MH1 SST Science 1049 MODIS-Aqua Level-3 mapped science-quality SST products
MPOC / MPIC NRT 1055 MODIS-Aqua Level-3 mapped NRT ocean color stream used for carbon products
MPOC / MPIC Science 1102 MODIS-Aqua Level-3 mapped science-quality carbon products

GCS path conventions

NRT products are published under satellite/<DATASET>_NRT/... paths, while science-quality products are published under satellite/<DATASET>/... paths. The exact product and period subdirectories vary by stream, but follow the same naming pattern used by the legacy ERDDAP-facing archive.

Example object prefixes:

satellite/MH1_NRT/chla/1day/
satellite/MH1/chla/1day/
satellite/MH1_NRT/sst/mday/
satellite/MH1/sstMask/8day/
satellite/MPOC_NRT/1day/
satellite/MPIC/mday/

SST masking

For both NRT and science-quality SST products, the workflow generates a masked SST file alongside the raw NASA download. The masking step reads the raw sst variable and the qual_sst quality variable, then sets SST values to fill where qual_sst < 0.

Pixels where qual_sst >= 0 are retained and written to an output variable named sstMasked. Raw SST files and masked SST files are published to separate configured GCS paths so ERDDAP can expose both the original NASA SST product and the CoastWatch masked variant.

Credentials and secrets

Earthdata Login credentials are provided to the Cloud Run job through Google Secret Manager. The job expects .netrc and URS cookie files to be available from Secret Manager mounts at runtime. The entrypoint copies these files into writable /tmp locations before the Python retrieval scripts run.

Expected runtime credential paths include:

/tmp/.netrc
/tmp/.urs_cookies

These files are runtime credentials only. They should not be committed to the source repository or included in the public Quarto documentation copy.
